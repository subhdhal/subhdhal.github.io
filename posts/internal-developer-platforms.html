Building Internal Developer Platforms: Golden Paths & Service Catalogs

November 2025

Introduction

In modern organizations with distributed microservices architectures, developer experience becomes a critical competitive advantage. Internal Developer Platforms (IDPs) are self-service tools that enable teams to provision infrastructure, deploy services, and manage their full lifecycle without waiting for platform engineers. In this post, I'll share insights from implementing an IDP that reduced service onboarding time by 70%.

What is an Internal Developer Platform?

An IDP is a curated layer of self-service capabilities built on top of your infrastructure. Instead of teams managing raw Kubernetes or AWS APIs, they interact with higher-level abstractions tailored to your organization's best practices.

Key components of an IDP include:

  • Service Catalog: Discoverable, self-documenting services and templates
  • Golden Paths: Opinionated, documented paths for common workflows
  • Developer Portal: Unified UI for discovering and managing services
  • Policy Enforcement: Automated guardrails ensuring security and compliance
  • Self-Service Infrastructure: Templated deployments reducing manual steps

The Problem We Solved

Before implementing our IDP, new microservices took 3-4 weeks to provision:

  • Teams submitted requests through ticketing systems
  • Platform engineers manually created infrastructure
  • Different teams followed different deployment patterns
  • Security and compliance reviews caused bottlenecks
  • Inconsistent observability and monitoring setups

Golden Paths: The Opinionated Way

Golden paths are curated, documented workflows representing best practices for common scenarios. Rather than every team reinventing the wheel, we created standardized paths for: 

# Example: Golden Path for Deploying a Python API Service

Inputs:
  - Service Name
  - Team Name
  - GitHub Repository URL
  - Port & Health Check Path

Automated Outputs:
  1. GitHub Actions CI/CD Pipeline (Docker build, tests, security scans)
  2. EKS Deployment (with resource limits, replicas, HPA config)
  3. AWS RDS Database (if needed)
  4. Application Monitoring (DataDog agent, logging)
  5. AWS IAM Roles (least-privilege access)
  6. DNS & Load Balancer Configuration
  7. Infrastructure Code (versioned, reviewable)

The result? Teams can go from zero to production-ready microservice in hours, not weeks.

Service Catalog: Discoverability at Scale

With 30+ microservices across teams, discoverability becomes a challenge. Our service catalog provides:

  • Metadata about each service (owner, slack channel, on-call runbooks)
  • Dependencies between services (visualization of service mesh)
  • Deployment status and health checks
  • Documentation links and API specifications
  • Cost allocation per service
  • Links to logs, metrics, and traces

Implementation: Self-Hosted Developer Portal

We built our internal developer portal using open-source tools, enabling: 

Technology Stack:
- Backstage (Developer Portal Framework)
- Terraform + Helm (Infrastructure Templates)
- GitHub Actions + ArgoCD (GitOps Workflows)
- Kubernetes (EKS Cluster)
- AWS Services (RDS, IAM, VPC, etc.)

Workflow:
1. Developer navigates portal and selects "Create New Service"
2. Fills in service details (name, team, repository)
3. Portal generates scaffolding code and infrastructure
4. GitHub repo created with CI/CD pipeline
5. Infrastructure provisioned via Terraform
6. Service appears in catalog
7. Developer can immediately deploy

Security & Compliance as Code

One major benefit: security and compliance are built-in, not bolted-on later.

  • Network Isolation: Every service gets proper VPC/subnet configuration
  • IAM Policies: Least-privilege roles generated automatically
  • Secrets Management: AWS Secrets Manager integration
  • Observability: DataDog agents and logging pre-configured
  • Scanning: SAST/DAST/SCA tools run in CI pipeline
  • Compliance: Audit trails and compliance reports auto-generated

Measuring Success

After implementing our IDP with golden paths and service catalogs:

  • 70% reduction in service onboarding time (3-4 weeks → 1-2 hours)
  • 3x faster deployment velocity across teams
  • 100% compliance with security baselines (no manual reviews needed)
  • 50% fewer platform engineer support tickets
  • 95% developer satisfaction with self-service experience
  • Standardized deployment patterns across 30+ microservices

Lessons Learned

  • Start simple: Golden paths for 80% use case, custom paths for 20%
  • Iterate based on feedback: Platform adoption drives feature prioritization
  • Document everything: Clear documentation reduces support burden
  • Version your infrastructure: Breaking changes should be communicated
  • Monitor usage: Track which paths are used, when they break
  • Make onboarding easy: New developers should understand platform in first day

Next Steps for Your Organization

If you're considering an IDP:

  1. Identify your most common deployment patterns
  2. Start with one golden path for your most frequent use case
  3. Gather feedback from developers
  4. Iterate and expand to more paths
  5. Consider open-source platforms like Backstage, Port, or OpsLevelwrap
  6. Invest in documentation and developer education

Conclusion

Internal Developer Platforms with golden paths and service catalogs transform how organizations scale engineering. By reducing toil and standardizing best practices, teams can focus on building features instead of managing infrastructure. The 70% reduction in onboarding time was just the beginning—the real value comes from enabling developers to ship secure, reliable code faster.
